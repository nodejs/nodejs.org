# Relatório de Segurança

Para mais detalhes sobre políticas de segurança ativas, faça check-out [nesta página](https://github.com/nodejs/node/security/policy).

## Reportando um erro no Node.js

Reporte erros de segurança no Node.js via [HackerOne](https://hackerone.com/nodejs).

O seu relatório será reconhecido dentro de 5 dias. e você receberá uma resposta mais
detalhada ao seu relatório no prazo de 10 dias, indicando os próximos passos em
manuseando seu envio.

Após a resposta inicial ao seu relatório. a equipe de segurança se esforçará para manter
você informado do progresso feito em direção a um anúncio fixo e completo,
e pode pedir informações ou orientações adicionais sobre o problema reportado
.

### Programa de recompensa do Node.js

O projeto Node.js envolve um programa oficial de recompensa para a segurança
pesquisadores e divulgações públicas responsáveis. O programa é gerenciado pela plataforma
HackerOne. Veja [https://hackerone.com/nodejs](https://hackerone.com/nodejs) para mais detalhes.

## Relatar um bug em um módulo de terceiros

Os erros de segurança em módulos de terceiros devem ser reportados aos respectivos mantenedores
.

## Política de divulgação

Aqui está a política de divulgação de segurança para Node.js

- O relatório de segurança é recebido e é atribuído um manipulador primário. Essa pessoa coordenará o processo de correção e lançamento. O problema é confirmado e uma lista de todas as versões afetadas é determinada. O código é auditado para encontrar quaisquer problemas similares potenciais. As correções são preparadas para todas as versões que ainda estão em manutenção. Essas correções não são enviadas para o repositório público, mas sim mantidas localmente aguardando o anúncio.

- Uma data de embargo sugerida para esta vulnerabilidade é escolhida e uma CVE (Comuns
  Vulnerabilidades e Exposições (CVE®️)) é solicitada para a vulnerabilidade.

- Na data do embargo, é enviada uma cópia do anúncio
  à lista de segurança do Node.js. As alterações são enviadas para o repositório público e novas compilações
  são implantadas para nodejs. rg. Dentro de 6 horas da lista de e-mails sendo notificada a
  uma cópia do consultório será publicada no blog do Node.js.

- Normalmente a data do embargo será definida para 72 horas a partir do momento em que o CVE é
  emitido. No entanto, isso pode variar dependendo da gravidade do bug ou da dificuldade
  na aplicação de uma correção.

- Este processo pode levar algum tempo, especialmente quando a coordenação é necessária
  com os mantenedores de outros projetos. Todos os esforços serão feitos para lidar com o bug
  da maneira mais oportuna possível; no entanto, é importante que sigamos
  o processo de lançamento acima para garantir que a divulgação seja tratada de uma maneira
  consistente.

## Recebendo atualizações de segurança

Notificações de segurança serão distribuídas pelos seguintes métodos.

- [Google Group](https://groups.google.com/group/nodejs-sec)
- [Blog do Node.js](/blog)

## Comentários sobre esta política

Se você tiver sugestões sobre como esse processo pode ser melhorado, envie uma
[pull request](https://github.com/nodejs/nodejs.org) ou
[file an issue](https://github.com/nodejs/security-wg/issues/new) para discutir.

A Fundação de Segurança de Código Aberto (OpenSSF) \[Emblema de Melhores Práticas]\(https://github. om/coreinfraestrutura/best-practices-badge) é uma maneira para projetos Free/Libre e Software de Código Aberto (FLOSS) mostrar que eles seguem as melhores práticas. Os projectos podem voluntariamente auto-certificar como seguem cada uma das melhores práticas. Os consumidores do selo podem rapidamente avaliar quais os projectos FLOSS que estão a seguir as melhores práticas e, como resultado, são mais susceptíveis de produzir software seguro de maior qualidade.
